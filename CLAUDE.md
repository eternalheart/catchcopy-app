## 0. 最重要原則（必須）

すべての実装は以下を満たすこと。1 つでも満たさない場合は未完了とする。

### 再利用可能な構成にすること
- ソースコードの可読性を最優先にする。
- UI、状態管理、API、データアクセス、ユーティリティ、ドメインロジックを責務ごとに分離する。
- 特定の画面や一時的な要件に依存しすぎない構成にする。

### コメントを出力すること
- 主要なクラス、関数、型、変数には説明コメントを入れること。
- 関数には目的、引数、戻り値、例外条件が分かるコメントを入れること。
- 複雑な if 分岐や条件判定には、なぜその条件が必要か分かるコメントを入れること。
- 自明な処理に過剰なコメントを入れず、保守に役立つコメントを優先すること。

### 型安全であること
- TypeScript は strict を有効にする。
- any は原則禁止する。
- unknown を使う場合は、型ガードまたはバリデーションで安全に扱う。
- 外部入力、API レスポンス、DB レコード、環境変数は型定義またはバリデーションを通す。
- 型エラーを無理に回避する実装をしない。

### エラーハンドリングされていること
- 処理上問題のないエラーは安全に無視してよい。
- クリティカルなエラーは、原因が分かるメッセージを UI またはログに出力する。
- 例外を握りつぶさない。
- ユーザーに見せるエラーと、開発者向けログを適切に分ける。
- セキュリティ上問題のある詳細情報を UI に表示しない。

### 商用利用しても著作権的に問題がない素材のみを使用すること
- アイコン、画像、フォント、音声、サンプルデータなどはライセンスを確認する。
- 権利関係が不明な素材を使用しない。
- 外部ライブラリを追加する場合は、ライセンスと必要性を確認する。

### README.md を Claude Code が必ず作成すること
- README.md には、実装内容、ローカル実行手順、Vercel へのデプロイ手順、（DB を使う場合は）Vercel Marketplace 経由のデータベースセットアップ手順を記載する。
- Claude Code がリポジトリ直下に README.md を出力すること。

## 1. 開発環境

以下の環境で開発される可能性があるため、特定 OS に依存しない実装・手順にすること。

- **OS**: Windows / Linux / macOS
- **エディタ**: VSCode
- **言語**: TypeScript（strict）
- **フレームワーク**: Next.js（App Router）
- **UI**: React Function Components + Hooks
- **Node.js**: v24.1.0 以上を前提
- **パッケージ管理**: プロジェクト作成時に選択したパッケージマネージャーを使用する。npm / pnpm / yarn のいずれかに統一する。複数の lock ファイルを混在させない。
- **デプロイ**: Vercel
- **データベース**: 必要な場合は Vercel Marketplace から接続した Postgres を使用する。新規構築では旧 Vercel Postgres を前提にしない。代表例として Neon Postgres を使用できる構成にする。

---

## 2. OS 非依存の実装・手順

Windows / Linux / macOS のいずれでも動作するように、以下を守ること。

- 特定 OS 専用のシェル構文を前提にしない。
- README.md のコマンド例は、可能な限り OS 共通で使える形式にする。
- パス区切り文字に依存する実装をしない（Node.js 側では `path.join`、`path.resolve` を使用。文字列結合でパスを作らない）。
- ファイル名やテキストは UTF-8 を前提に扱う（日本語ファイル名や日本語文字列を扱っても壊れないように）。
- 改行コードの違いで動作が変わらないようにする。
- 環境変数の設定方法が OS によって異なる場合は、README.md に OS 別の補足を記載する。
- 開発用スクリプトは package.json の scripts に集約し、OS 差異を吸収する。

## 3. 技術構成

Next.js + Vercel 前提で以下の構成にすること。

- Next.js App Router を使用する。
- `src/` 配下に実装する。
- 画面は可能な限り Server Component を基本とする。
- ブラウザ API が必要な箇所のみ Client Component にする。
- API が必要な場合は Route Handler を使用する。
- フォーム送信やサーバー側処理には、要件に応じて Server Action または Route Handler を使用する。
- 外部 API・DB アクセスはサーバー側に閉じる。ブラウザから API キーや DB 接続文字列を参照させない。
- 環境変数は `.env.local` を使用する。`.env.local` は Git 管理しない。
- Vercel 上では Project Settings または Storage Integration により環境変数を管理する。
- 設定値、定数、型定義、バリデーション、データアクセス、UI を適切に分離する。
- 外部サービスとの通信は、エラー、タイムアウト、不正レスポンスを考慮する。

## 4. UI/UX 要件

- レスポンシブ対応。デスクトップブラウザで問題なく利用できること。
- タブレットやスマートフォンでも表示が崩れないことが望ましいが、スマートフォン最適化は必須ではない。
- 処理中ローディング表示を行うこと。
- エラー表示を行うこと。
- 操作完了時に必要なフィードバックを表示すること。
- 入力フォームがある場合は、入力値のバリデーションを行うこと。
- 画面の状態が分かりにくくならないようにすること。

## 5. データ保存・外部APIキー方針

- 外部 API キー・DB 接続文字列などの秘密情報はサーバー側のみで扱う（Route Handler または Server Action）。
- Client Component から秘密情報を直接参照しない。
- 秘密情報は環境変数で管理する。`.env.local` をローカル開発用に使用し、`.gitignore` に含める。
- 本番環境の環境変数は Vercel Dashboard で管理する。
- 環境変数の値をログに出力しない。
- クライアントに公開してよい値のみ `NEXT_PUBLIC_` を付ける。API キーや DB 接続文字列に `NEXT_PUBLIC_` を付けない。
- DB を使う場合は ORM/クエリビルダーで型安全に扱う（推奨 Prisma または Drizzle）。大容量バイナリは DB に保存しない。

## 7. 実装ルール

- TypeScript の any は原則禁止。unknown は型ガードまたはバリデーションで安全に扱う。
- React の状態管理は責務ごとに分離する。
- UI、API、外部サービス、ドメインロジック、ユーティリティを分離する。
- Server Component と Client Component の責務を明確にする。Client Component は必要な箇所に限定する。
- 主要なクラス、変数、関数にはコメントを入れる。
- 例外は握りつぶさず、分かりやすいエラーとして返す。
- 商用利用上問題のないアイコン、絵文字、画像、フォント、外部素材のみを使用する。
- 外部ライブラリ追加時は、ライセンスと必要性を確認する。
- セキュリティ上重要な処理はサーバー側で行う（API キー・認証・機密情報の処理をクライアント任せにしない）。
- 入力値はサーバー側でも必ず検証する。
- API レスポンスは適切な HTTP ステータスコードを返す。エラーレスポンスに機密情報を含めない。
